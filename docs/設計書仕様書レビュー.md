# MarkTeXset 設計書・仕様書レビュー

レビュー対象:

- [MathMD_Specification.md](./MathMD_Specification.md)
- [MathMDの設計案.md](./MathMDの設計案.md)

## 総評

TeX相当の活版品質を、Pandoc の Markdown→LaTeX 出力を TeX エンジンで組版した PDF を基準にする方針は明確になった。品質目標の方向性もよい。

一方、現状は「企画としては十分だが、実装仕様としては未確定」である。特に YAML スキーマ、数式処理系、ラベル・相互参照・参考文献の扱いに文書間の矛盾があり、実装者によって異なる解釈が生じる。

## 最優先で解決すべき問題

### 1. YAML のルート構造が矛盾している

仕様書では `mathmd` 配下に設定を書く。

`MathMD_Specification.md:55`

```yaml
mathmd:
  import: ...
  meta: ...
  layout: ...
  command: ...
```

一方、設計書の import 例では `import`、`layout`、`math` をトップレベルに置いている。

`MathMDの設計案.md:131`

```yaml
import:
layout:
math:
```

通常の文書例では `mathmd.command` を使っているため、どちらが正式仕様か統一が必要である。

推奨形式:

```yaml
---
mathmd:
  import:
    - presets/base.yaml
  meta:
    title: ...
  layout:
    ...
  command:
    ...
---
```

import ファイルだけ `mathmd` を省略する設計にする場合は、その例外を明記する。

### 2. `layout` のスキーマが複数存在する

仕様書では次の形式である。

```yaml
layout:
  size: A4
  margins: 25mm
  numbering:
    headings: true
```

設計書では次の形式が使われている。

```yaml
layout:
  class: article
  paginate: true
  page:
    size: A4
    margins: 25mm
```

さらに本文では以下のキーも使われる。

```text
layout.heading.visible
layout.heading.h1
layout.footnote
```

`size` と `page.size`、`numbering.headings` と `heading.visible` のどちらを正式仕様にするか決める必要がある。

### 3. 数式エンジンの方針が不一致である

仕様書は「KaTeX Subset」としている。

`MathMD_Specification.md:193`

設計書は「MathJaxベース」としている。

`MathMDの設計案.md:318`

また、可換図式について `tikz-cd`、`tikzjax`、`tikzcd` フェンス、`cd` フェンスが混在している。

最低限、次を確定する必要がある。

- 数式レンダラーは KaTeX か MathJax か
- HTML と PDF で同じレンダラーを使うか
- 数式番号、`\label`、`\ref` を誰が処理するか
- 図式の正式な記法は ` ```tikzcd ` か ` ```cd ` か
- tikz-cd を初期リリースでサポートするか

初期リリースでは tikz-cd を非目標にする判断も可能である。

### 4. Pandoc baseline の比較条件が抽象的である

品質の方向性は明確だが、基準 PDF の条件が定義されていない。

`MathMD_Specification.md:11`

少なくとも以下を固定した方がよい。

- Pandoc のバージョン
- 使用する TeX エンジン
- document class
- フォント
- 用紙サイズと余白
- 数式パッケージ
- bibliography の処理方式
- 比較対象となるサンプル文書

「標準的な TeX エンジン」では再現性がない。例えば `LuaLaTeX + article + geometry` のような基準プロファイルを定める必要がある。

## 重要な仕様上の曖昧さ

### 5. 参考文献の扱いが一致していない

仕様書では見出し属性を使う。

`MathMD_Specification.md:147`

```markdown
# 参考文献 {.references}
```

設計書では HTML コメントを使う。

`MathMDの設計案.md:272`

```markdown
<!--- refs: References -->
```

さらに、参考文献データを読み込むだけで、本文中の引用記法が定義されていない。

例えば次の記法を採用するのか決める必要がある。

```markdown
[@doe2020]
```

BibLaTeX 相当の処理をするのか、Pandoc citeproc を使うのか、文献リストだけを出力するのかも明記する。

### 6. ラベル・参照の記法が二系統ある

仕様書は標準 Markdown リンクである。

```markdown
[定理1.2](#thm:main)
```

設計書は `@hoge` と `[...](@hoge)` を使う。

`MathMDの設計案.md:274`

`@hoge` がラベルを置く記法なのか参照記法なのかが曖昧である。

TeX に寄せる場合でも、ラベルの設置と参照の記法を分けると明確になる。

```markdown
定理本文。 {#thm:main}

定理 [定理1.2](#thm:main) を使う。
```

「定理1.2」の自動補完は後続仕様に分離するのが安全である。

### 7. マクロ定義の形式が一致していない

仕様書には Arrow-style が登場する。

`MathMD_Specification.md:84`

```yaml
abs: (1) -> \left| #1 \right|
```

設計書では構造化形式である。

`MathMDの設計案.md:211`

```yaml
abs:
  args: 1
  body: '\left|#1\right|'
```

構造化形式に統一するのがよい。Arrow-style は `#` が YAML のコメント開始文字になるため、そのままでは正しく解釈できない。

### 8. import のマージ規則が未確定である

仕様書のコメントには、リストについて「置換（または追加、実装依存）」とある。

`MathMD_Specification.md:61`

一方、設計書ではリストは置換と明記されている。

`MathMDの設計案.md:176`

「実装依存」は仕様書から除くべきである。特に次を明記する。

- `bibliography` は置換か追加か 
- `import` 自体は継承するか
- map は浅いマージか深いマージか
- import 間の同名マクロは常にエラーか
- ルート文書だけが import を上書き可能か

### 9. YAML サンプルに構文上の問題がある

著者情報の例では、`url` や `affiliation` のインデントが著者要素の内部になっていない例がある。

`MathMDの設計案.md:103`

また、import ファイル例では正式なキー `command` ではなく `math` が使われている。

`MathMDの設計案.md:144`

仕様書のサンプルは、すべてそのまま YAML パーサで解釈できる状態にするべきである。

## 実装前に追加したい仕様

### 10. Markdown 方言の定義

`CommonMark (GFM)`、Pandoc Markdown、PHP Markdown Extra の機能が混在している。

以下の対応範囲を明記した方がよい。

- 表
- タスクリスト
- 生 HTML
- HTML コメント
- 複数行数式
- YAML Frontmatter
- Heading Attributes
- 定義リスト
- Callout

正式には「MarkTeXset Markdown 方言」として対応表を作るとよい。

### 11. 改ページ記法の衝突

本文中の `---` を改ページに使うと、Markdown の水平線と衝突する。

`MathMD_Specification.md:168`

以下のような専用記法の方が安全である。

```markdown
<!-- pagebreak -->
```

または:

```html
<div class="page-break"></div>
```

`---` を改ページ専用にする場合は、水平線としては使えないことを明記する。

### 12. 定理環境の意味論が不足している

現在は見た目の構文が中心で、以下が未定義である。

- 定理番号の単位
- 章ごとの番号リセット
- theorem / lemma / proposition が同じ番号系列か
- definition / example を番号付けするか
- proof に番号を付けないこと
- Q.E.D. の表示位置
- 定理内の見出し、段落、数式の扱い
- `{#id}` を callout のどこに付けるか

「任意のクラス名を受け付ける」と「サポートされる環境を列挙する」は分けて定義した方がよい。

### 13. エラー仕様が不足している

行番号、列番号、修正候補は定義されているが、次が未定義である。

- warning と error の区別
- エラー時に PDF を生成するか
- 複数エラーをまとめて報告するか
- 未解決参照や重複 ID の扱い
- YAML 構文エラーの形式
- unsupported TeX command の扱い
- 列番号の単位（Unicode code point / UTF-16 / byte）

## 結論

品質目標の方向性はよい。Pandoc→TeX→PDF を最低基準としたことで、「TeX相当」という曖昧な表現は大きく改善された。

ただし、現状のままでは実装仕様としては未確定である。特に YAML、数式処理、参照系の3点を統一すれば、実装に移せる仕様書に近づく。

## 旧解決案（統合前・参照用）

### 1. YAML のルート構造が矛盾している

import する yaml は以下の形式にする:

### 2. `layout` のスキーマが複数存在する

以下を採用する:
```yaml
---
  layout:
    size: A4
    margins: 25mm
    paginate: true
    heading:
      numbered: true # 章番号の自動採番
      h1: "Chapter {h1.arabic}." # 番号レイアウト
      h3: "{h1.roman}.{h2.roman}.{h3.roman}"
      # `Alph`, `alph`, `Roman`, `roman`, `arabic`, `fnsymbol` をサポート
      # 個別に省略したい場合は `#:` や `##:` などにする
    theorem:
      definition: # 文中で呼び出すコマンド名 (自由に設定するもの)
        numbered: true # 番号の自動採番
        title: "Definition {definition.arabic}." # タイトルテキスト
        style: "definition" # 定理スタイル (TeX の `\theoremstyle{}` 準拠)
      theorem:
        numbered: true
        title: "Theorem {theorem.arabic}."
        style: "plain"
      proof:
        numbered: false
    equation:
      numbered: false # 数式環境の番号の自動採番のデフォルト (個別の数式番号は数式ブロックで指定する)
---
```

`LuaLaTeX + article` 準拠とする

用紙サイズと余白もLuaLaTeXのデフォルトとし、フロントマッターで調整できるとする。

## 解決案（正式決定版）

以下は、grill-me セッションで確定した設計判断である。旧解決案と矛盾する場合は、本節を優先する。

### 1. YAML と import

- ルート文書は Frontmatter なしでも許可し、既定値で処理する。
- Frontmatter は文書先頭に一度だけ置く。
- ルート設定は mathmd 配下に置く。
- import ファイルは Frontmatter 区切りを持たず、mathmd の設定内容だけを持つ YAML マップとする。
- import ファイルの再帰 import、mathmd 再ラップ、未知キー、YAML 内の重複キーはエラー。
- ルート Frontmatter の未知キーと重複キーもエラー。
- import パスは入力 Markdown のディレクトリ基準。存在しない場合はエラー。
- 絶対 import パスは許可するが、../ による親ディレクトリ参照は禁止。
- 同一ファイルの重複 import は最初の位置で一度だけ処理し、後続指定は警告して無視する。
- 通常の設定値は後勝ち、bibliography と network.domains は追加マージする。
- import 間のマクロ・演算子名衝突はエラー。
- ルート文書は import のマクロ・演算子を上書きできる。

### 2. Frontmatter と既定値

フェーズ1の主要な既定値は次の通り。

~~~yaml
mathmd:
  meta:
    language: en
    timezone: UTC
  citation:
    style: numeric
    heading:
      text: References
      level: 2
  layout:
    class: article
    size: A4
    margins: 25mm
    paginate: true
    page:
      number:
        visible: true
        position: bottom-center
        format: "{page.arabic}"
      style: plain
    title:
      date-format: yyyy-mm-dd
    heading:
      numbered: true
      numbering-depth: 3
      toc-depth: 3
      h1: "{h1.arabic}."
      h2: "{h1.arabic}.{h2.arabic}."
      h3: "{h1.arabic}.{h2.arabic}.{h3.arabic}."
    counter:
      max-depth: 3
    footnote:
      format: "{footnote.arabic}"
      placement: bottom
    equation:
      numbered: false
      display: "({equation.arabic})"
      reference: "式 ({equation.arabic})"
~~~

- layout.class はフェーズ1では article のみ。未指定時も article、それ以外はエラー。
- meta.language はフェーズ1では en / ja のみ。未指定時は en。最終フェーズでは BCP 47 を受け付け、未対応言語は en にフォールバックする。本文からの自動言語判定は行わない。
- meta.timezone は IANA 名または UTC。既定値は UTC、不正値はエラー。
- meta.date 未指定時は指定タイムゾーンのシステム時刻を使う。
- layout.title.date-format のフェーズ1の許可トークンは yyyy、mm、dd。未知トークンはエラー。
- 用紙サイズは A4 のみ。余白は mm、cm、in、pt を許可し、負の値はエラー、ゼロは許可する。map の省略辺は 25mm とする。

### 3. 見出し・目次・カウンター

- 見出し番号の既定形式は 1.、1.1.、1.1.1.。
- numbering-depth: 3 を超える見出しは表示するが番号を付けない。
- toc-depth: 3 を超える見出しは表示するが目次に含めない。
- #: のような番号なし見出しを許可する。番号なし見出しはカウンターを進めない。
- 番号なし h1 の下位見出しは欠けた階層を 0 として表示する（例: 0.1. 背景）。
- 目次要素は <maketoc />。本文側で深さを上書きする機能は持たない。
- 目次は見出し番号・タイトル・ページ番号を表示する。複数指定時は最初だけ有効、後続は警告する。
- 対象見出しがない場合は警告して空の目次を生成する。
- 自動生成される参考文献見出しも通常の見出しとして番号・目次の対象とする。
- カウンタースコープは {h1 > h2 > theorem.arabic} のように任意個を指定できる。
- スコープ要素数は layout.counter.max-depth 以下（既定値 3）。超過はエラー。
- 番号形式は arabic、roman、Roman、alph、Alph。未知形式・未知カウンターはエラー。
- タイトル中のリテラル波括弧は {{ / }} でエスケープする。

### 4. Callout / 定理環境

設定キーは layout.callouts とする。環境名と表示 style は独立して設定できる。

~~~yaml
layout:
  callouts:
    theorem:
      title: "Theorem {theorem.arabic}."
      style: plain
    lemma:
      title: "Lemma {theorem.arabic}."
      style: plain
    definition:
      title: "Definition {definition.arabic}."
      style: definition
    proof:
      title: Proof
      style: proof
~~~

- numbered キーは使用しない。title にカウンタープレースホルダーがあれば番号付き、なければ番号なし。
- 登録済み環境で title を省略した場合は環境名をタイトルに使う。
- style を省略した場合は plain。
- 未登録の [!foo] は title: foo、style: plain、番号なし。
- 入力タイトルは YAML の title の後ろに付加する。例: Theorem 1. ピタゴラスの定理。
- フェーズ1の style は plain、definition、remark、proof。未知 style は警告して plain にフォールバックする。
- 最終フェーズでは組み込み style と任意 CSS class の両方を許可する。
- Callout 環境名は大文字小文字を区別する。
- proof は番号なし、YAML の title を使い、末尾に □ を自動付与する。明示的な \qed の有無は検査しない。
- Callout 内部では複数段落、インライン数式、ブロック数式、箇条書き、脚注を許可する。入れ子 Callout、表、画像、コードブロック、Callout 内の脚注定義はフェーズ1では禁止する。

### 5. ID・相互参照・参考文献

- 数式 label、Markdown の {#id}、Callout の ID は同じ名前空間を共有する。
- 重複 ID はエラー。
- ID を明示しない見出しには自動 ID を付けず、参照対象にもしない。
- 数式内の参照先は \label{eq:quadratic} で定義する。
- 内部参照は #id の標準 Markdown fragment link に統一する。@id は採用しない。
- [式](#eq:quadratic) は入力テキストをそのまま表示し、[](#eq:quadratic) は参照先に応じて自動表示する。
- 数式の空参照は 式 (1)、定理・定義は YAML title の番号部分、番号なし環境は入力タイトルを表示する。
- 未解決参照は警告。空参照は赤太字で (eq:unknown)、明示テキスト付きは赤太字で 式 (eq:unknown) のように表示する。

文献引用は Pandoc 互換とする。

~~~markdown
[@doe2020]
[@doe2020; @smith2021]
~~~

- citation style はフェーズ1では numeric のみ。未指定時も numeric、未対応値はエラー。
- 番号は本文で最初に引用された順、表示は [1]、複数引用は番号順。
- 連続番号の範囲表記は後続フェーズ。
- 同じ key は同じ番号を再利用する。
- 同一引用グループの重複 key は警告して除去する。
- 未解決 key は警告。単独なら赤太字の [missing2020]、複数引用の一部なら引用全体を赤太字にする。
- citation key は大文字小文字を区別する。候補があれば Did you mean ...? を表示する。

参考文献リストは <references /> の位置に生成する。最初の1つだけ有効、後続は警告。自動見出しは citation.heading.text / citation.heading.level で設定し、フェーズ1の既定値は References / level 2。文献データがない場合は警告して見出しだけ生成する。文献データがあっても <references /> がなければ何も出力しないが、本文中に引用があるのに <references /> がない場合はエラーとする。

.bib は Markdown ファイル基準のローカルパスのみ許可する。../、絶対パス、URLは禁止。不存在・構文不正はエラー、重複 key は警告して後勝ち。複数ファイルは指定順、同一ファイルは登場順で後勝ちとする。

### 6. Markdown 方言（フェーズ1）

本文全体で CommonMark の基本構文をサポートする。ただし生 HTML は禁止し、HTML コメントは自由記述として許可する。Callout 内部は専用の許可要素に限定する。

フェーズ1の追加機能は脚注、MathJax 数式、Callout、Markdown テーブル、YAML Frontmatter、Heading Attributes の {#id} / {#id .class}、Markdown の水平線 ---。

フェーズ1では取り消し線、タスクリスト、定義リスト、include、embed、figure、画像サイズ拡張、任意の生 HTML は非対応。定義リストは最終フェーズまでに TeX の description 環境相当として再設計する。

Heading Attributes の class 名は任意の CSS 識別子を複数指定可能。CSS に定義がない class は何もしない。最終フェーズでは任意の HTML 属性を検討する。

テーブルセルはインライン Markdown（数式、強調、リンク、インラインコード）まで許可し、ブロック要素は非対応。左・中央・右揃えを GFM 互換でサポートする。

コードブロックはフェンス付き・インデント付きともにサポート。言語指定は保持するが、ハイライト・行番号なし。Callout 内では非対応。プレビューでは長い行を横スクロールし、PDF ではページ幅超過を警告してそのまま出力する。

画像は表示のみ、ローカルファイル限定。Markdown ファイル基準、../・絶対パス・URLは禁止。不存在はエラー。PNG、JPEG、SVG を許可し、SVG 内スクリプトは拒否する。空の代替テキストにはファイル名を使い、画面・PDFには表示せず alt 属性だけに使う。キャプション・図番号・相互参照は後続フェーズ。

### 7. 専用要素とページ制御

HTML コメントは制御に使わない。専用要素は自己完結形式、トップレベルのブロックのみ許可する。

~~~html
<pagebreak />
<pagestyle name="empty" />
<maketoc />
<maketitle />
<references />
~~~

専用要素の未知属性、未知 HTML 要素、必須属性の欠落はエラー。pagestyle の name は必須で、empty、plain、headings のみ許可。未知値は警告して plain にフォールバックする。

<pagebreak /> は先頭・末尾・連続指定でも常に実行し、空白ページごとに警告する。空白ページもページ番号を消費する。

<maketitle /> は title、著者、所属、日付をまとめて生成する。複数指定時は最初だけ有効、後続は警告。meta.title がない場合は警告して生成しない。著者は一人ごとに改行、所属と URL を表示し、URL はリンク化する。日付は layout.title.date-format に従う。

### 8. 数式と TeX subset

数式エンジンは MathJax。HTML と PDF で同じ SVG 出力を使い、アクセシビリティ用 MathML も併記する。数式フォントは Latin Modern Math 相当。

インライン数式は $...$、ブロック数式は $$...$$。\\ と & / && はブロック数式内だけで許可し、インラインではエラー。

$$...$$ は独自に複数行をサポートする。行区切りは \\ のみ、実改行では行分割しない。& は位置合わせアンカーで、出現位置ごとに左右揃えを交互に切り替える。連続 & は任意個数を許可し、個数分だけ位置を進める。行ごとのアンカー数が異なる場合は不足分を空セルとする。空行は無視し、末尾 \\ は警告して除去する。

番号は行単位。equation.numbered: true は全行を番号付けするが、\notag の行は番号なし・カウンター非加算。equation.numbered: false は \label のある行だけ番号付けする。\notag と \label の同一行はエラー。単一行の \notag も許可する。

行列環境内の & / \\ は列・行区切り、外側は独自複数行構文とする。行列環境の入れ子は禁止。

許可する行列系環境は cases、matrix、pmatrix、bmatrix、Bmatrix、vmatrix、Vmatrix、smallmatrix。aligned、alignedat、align、align*、gather、gather* は非対応。

\text、\operatorname、\frac、\sqrt、主要な大型演算子、\mathbb、\mathcal、\mathrm 等を許可する。\operatorname は YAML 演算子定義と独立。コマンド・環境名は大文字小文字を区別する。

\text{...} は Unicode、プレーンテキスト、\%、\#、\$、\&、\_、\{、\} のエスケープだけを許可する。\text の入れ子、数式コマンド、\textbackslash は禁止する。

マクロは args / body の構造化形式。呼び出しは \newcommand 互換で、単一トークンは {} なし、複数トークンは {...}。引数数は 0〜4、オプション引数は非対応。\newcommand、\renewcommand、\def、catcode 操作、ページ制御、未定義コマンドはエラー。

演算子は operators に統一し、limits 未指定時は false。font は mathrm、mathcal、mathbb のみ、未指定時は operatorname 相当のローマン体。最終フェーズでmathbf、mathsf、mathttを追加する。

組み込み TeX コマンド・演算子の上書きは、定義に redef: true がある場合だけ許可。既定値は false。ユーザー定義同士、マクロと演算子の衝突は常にエラー。redef は既存の組み込み名にだけ適用できる。

### 9. 脚注

番号は arabic、各ページ下部に配置する。同一脚注ラベルの重複定義はエラー。フェーズ1では Callout 内部に脚注定義を置けない。未定義参照は警告し、[^missing] 全体を赤太字で表示する。脚注番号・リンクは生成せず、未定義参照は番号を消費しない。

### 10. 診断と終了コード

エラーがある場合は可能な限り全エラーを収集して一括報告し、HTML/PDF は生成しない。警告の場合は生成する。

診断メッセージは英語固定。既定は人間向けテキスト、--format json は構造化 JSON。JSON は stdout、進捗ログは stderr。診断がない場合も {"diagnostics":[]} を出力する。

診断 JSON の必須フィールドは severity、code、message、location.file、location.start、location.end。suggestions は必要時に持つ。位置は 1 始まりの Unicode code point、end は排他的な半開区間。ファイルパスはプロジェクトルートからの相対パス。import エラーは本文側の指定位置を主位置とし、参照先を relatedLocations に含める。診断は error → warning、各グループ内は文書位置順。

候補提示は、マクロ・演算子・組み込み TeX コマンドを同じ名前空間、環境名・citation key を別系統として扱う。編集距離 2 以下、最大 3 件、距離順・同距離は辞書順。

終了コードは、成功または警告のみ 0、エラー 1、CLI 使用方法エラー 2。

### 11. フォント・ネットワーク・レンダリング

既定フォントは欧文 Latin Modern、日本語 Harano Aji、数式 Latin Modern Math 相当。フォントは配布パッケージに同梱し相対パスで読み込む。読み込み失敗はエラー。

PDF は Playwright 管理下の Chromium で生成し、システムブラウザは使わない。Playwright、同梱 Chromium revision、Paged.js、MathJax は lockfile で固定する。Pandoc baseline は Pandoc・LuaLaTeX・TeX Live 等を含む Docker イメージを digest 固定し、更新は回帰テスト通過後の明示的な更新 PR とする。

外部ネットワークはデフォルト禁止。許可する場合は root Frontmatter の network.allow: true と許可ドメインが必須。network.domains は import から設定でき、追加マージする。サブドメインは明示、HTTPS のみ。画像・CSS・フォントは取得可能だが、JavaScript、iframe、embed、Markdown include は常に禁止。bib は常にローカル限定。

外部リソースは URL と内容ハッシュをキャッシュし、内容変更時は警告して新しい内容で生成する。URL とハッシュをリセットする明示オプションを用意する。

### 12. TeX baseline とテスト

比較用 baseline は文書言語で切り替える。

~~~text
en: LuaLaTeX + article + Latin Modern
ja: LuaLaTeX + ltjsarticle + luatexja + Harano Aji
~~~

MarkTeXset 自体は TeX でコンパイルしない。TeX は比較用のみ。

受け入れテストは正常系・異常系・レイアウト系に分割し、1 機能 1 文書とする。

- 正常系: HTML の意味的 DOM 構造と主要要素 assertion
- 異常系: 期待診断 JSON と終了コード
- レイアウト系: PDF のページ数、テキスト位置、はみ出し、重なり

診断 JSON は severity、code、位置情報を完全一致し、message は部分一致、未知フィールドは許可する。PDF の画像差分率は合否に使わず参考情報として記録する。テスト時は meta.date を明示し、必要に応じてタイムゾーンと時計も固定する。
